# BizFlow — Проектный контекст

## Обзор проекта

**BizFlow** — централизованная система учёта бизнеса, построенная на PHP 8.4 и фреймворке Yii3.

### Назначение

Система предоставляет единый, достоверный контур учёта для:
- Товаров и остатков
- Движения товаров
- Сделок и клиентов
- Точек продаж
- Выручки и аналитики
- Документов по сделкам
- Интеграций с внешними системами (Эвотор, API онлайн-кассы)

### Технологический стек

| Компонент | Технология |
|-----------|------------|
| Язык | PHP 8.4 |
| Фреймворк | Yii3 |
| База данных | MySQL 8 |
| Кэширование | Redis 7 |
| Контейнеризация | Docker + Docker Compose |
| Web-сервер | Caddy |

## Архитектура

Проект следует принципам **DDD (Domain-Driven Design)**, **SOLID**, **DRY**, **KISS**.

### Структура слоёв

1. **Domain** — Entities, Value Objects, Aggregates, Domain Services, Domain Events
2. **Application** — Use Cases (1 сценарий = 1 класс), DTO, Interfaces (ports)
3. **Infrastructure** — DB (MySQL), Redis, API интеграции, реализации репозиториев
4. **Presentation** — REST API

### Структура каталогов

```
├── config/           # Конфигурации приложения (common, web, console)
├── src/              # Исходный код приложения
│   ├── Console/      # Команды CLI
│   ├── Database/     # Миграции БД
│   ├── Shared/       # Общие компоненты
│   └── Web/          # Веб-компоненты (контроллеры, действия)
├── tests/            # Тесты (Unit, Functional, Web, Console)
├── public/           # Публичная директория (index.php, assets)
├── docker/           # Docker-конфигурации (dev, test, prod)
├── docs/             # Документация
└── runtime/          # Временные файлы (кэш, логи)
```

### Autoloading

- `App\` → `src/`
- `App\Tests\` → `tests/`

## Правила разработки

### Стиль кодирования

- **Форматирование**: PHP CS Fixer с правилами `@PER-CS`
- **Отступы**: 4 пробела (согласно `.editorconfig`)
- **Конец строки**: LF
- **UTF-8**: кодировка файлов
- **Строгие типы**: `declare(strict_types=1)` в начале каждого PHP-файла

### Именование

| Элемент | Стиль | Пример |
|---------|-------|--------|
| Классы | PascalCase | `PageRepository` |
| Методы и свойства | camelCase | `findOneBySlug()` |
| Константы | UPPER_SNAKE_CASE | `ENVIRONMENT_DEV` |
| Файлы миграций | `MYYMMDDHHMMName` | `M251102141707Page` |

### Принципы проектирования

1. **Один Use Case = один класс** — без бизнес-логики в контроллерах
2. **Интерфейсы репозиториев** — в domain/application, реализация — в infrastructure
3. **Агрегаты** — одна транзакционная граница = один агрегат
4. **Идемпотентность** — все внешние операции должны быть идемпотентными
5. **UUID первичные ключи** — генерация ID в коде, не в БД (использовать `ramsey/uuid`)
6. **Неизменяемость (Immutability)** — Value Objects должны быть immutable
7. **Dependency Injection** — использовать constructor injection, авто-wiring через DI-контейнер

### Тестирование

- **Фреймворк**: Codeception 5.x
- **Направления тестов**:
  - `tests/Unit/` — юнит-тесты
  - `tests/Functional/` — функциональные тесты
  - `tests/Web/` — веб-тесты
  - `tests/Console/` — тесты CLI-команд
- **Покрытие**: включено, анализируется `src/*` и точки входа

### Статический анализ

- **Psalm**: уровень ошибок 1, кэш в `runtime/cache/psalm`
- **Rector**: поддержка PHP 8.2+ правил
- **Composer Dependency Analyser**: контроль используемых зависимостей

### Безопасность

- Валидация всех входных данных
- Защита от SQL injection (использовать query builder)
- CSRF-защита для форм
- RBAC для авторизации

### Git-практики

- Использовать `.gitignore` и `.gitattributes`
- **Не коммитить**: `runtime/`, `vendor/`, `.env`
- Миграции БД должны быть в VCS

## Бизнес-правила

### Источники данных

Система работает с **несколькими источниками данных**:

**Внешние:**
- Эвотор — API онлайн-кассы (чеки, продажи, позиции)

**Внутренние:**
- Сделки, создаваемые пользователем
- Счета и документы (включая B2B)
- Ручные операции (корректировки, движения)

**Важно:**
- Источники могут **дублировать данные**
- Источники могут **противоречить друг другу**
- Система обязана приводить данные к **согласованному состоянию**

### Остатки

- **Не могут быть отрицательными**
- Изменяются только через доменные операции

### Резервы

- Резерв **уменьшает доступный остаток**
- Зарезервированный товар **нельзя продать/отгрузить**
- Снятие резерва — **явная операция**

### Сделки

- Связывают товар, клиента и точку продаж
- Могут создавать движения товара

## Важные ограничения и запреты

### Запреты

- Не смешивать слои (domain/application/infrastructure/presentation)
- Не писать бизнес-логику в контроллерах
- Не использовать Active Record как доменную модель
- Не нарушать границы агрегатов
- Не делать «магические» решения без объяснения

### Приоритеты

1. Консистентность данных
2. Простота и читаемость
3. Предсказуемость поведения
4. Масштабируемость
5. Производительность

## Примечания

- Изменения в `config/configuration.php` требуют запуска `composer yii-config-rebuild`
- Домены и bounded contexts **не фиксируются заранее** — определяются на основе сценариев и инвариантов
- Межагрегатное взаимодействие происходит через application layer
- Все внешние операции должны быть **идемпотентными** с защитой от дубликатов
- Система должна поддерживать возможность **повторной обработки** операций
