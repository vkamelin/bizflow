## 1. ОБЩЕЕ ОПИСАНИЕ СИСТЕМЫ

BizFlow — это централизованная система учёта бизнеса:

* товары и остатки
* движение товаров
* сделки и клиенты
* точки продаж
* выручка и аналитика
* документы по сделкам
* интеграции с внешними системами

Цель — единый, достоверный контур учёта без рассинхронизации данных.

## 3. АРХИТЕКТУРНЫЙ ПОДХОД

Используется:

* DDD (Domain-Driven Design)
* SOLID
* DRY
* KISS

### 3.1. Разделение слоёв

1. Domain

   * Entities
   * Value Objects
   * Aggregates
   * Domain Services
   * Domain Events

2. Application

   * Use Cases (1 сценарий = 1 класс)
   * DTO
   * Interfaces (ports)

3. Infrastructure

   * DB (MySQL)
   * Redis
   * API интеграции
   * Реализации репозиториев

4. Presentation

   * REST API

(структура слоёв сохраняется, без жёсткой фиксации доменов заранее)

---

## 4. ДОМЕННОЕ МОДЕЛИРОВАНИЕ

* домены и bounded contexts **не фиксируются заранее**
* определяются на основе сценариев и инвариантов

Правила:

* агрегаты выделяются только при наличии инвариантов
* одна транзакция = граница агрегата
* межагрегатное взаимодействие через application layer

---

## 5. ИСТОЧНИКИ ДАННЫХ

Система работает с несколькими источниками:

### 5.1. Внешние

* Эвотор — API онлайн-кассы (чеки, продажи, позиции)

### 5.2. Внутренние

* сделки, создаваемые пользователем
* счета и документы (включая B2B)
* ручные операции (корректировки, движения)

Особенности:

* источники могут дублировать данные
* источники могут противоречить друг другу
* система обязана приводить данные к согласованному состоянию

---

## 6. ОСНОВНЫЕ БИЗНЕС-ПРАВИЛА

### Остатки

* не могут быть отрицательными
* изменяются только через доменные операции

### Резервы

* резерв уменьшает доступный остаток
* зарезервированный товар нельзя продать/отгрузить
* снятие резерва — явная операция

### Сделки

* связывают товар, клиента и точку продаж
* могут создавать движения товара

---

## 7. ТРЕБОВАНИЯ К ПРОЕКТИРОВАНИЮ

### 7.1. Агрегаты

* один агрегат = одна транзакционная граница
* внутри агрегата — строгая консистентность
* межагрегатное взаимодействие через application layer

### 7.2. Use Cases

* один сценарий = один класс
* без бизнес-логики в контроллерах
* без прямого доступа к БД

### 7.3. Репозитории

* только интерфейсы в domain/application
* реализация в infrastructure

* бизнес-логика вне контроллеров
* инфраструктура изолирована

---
